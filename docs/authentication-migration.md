# Authentication Migration (Auth0 to BetterAuth)

This document outlines the transition from Auth0 to BetterAuth, the database changes, and the administrative tools available for managing users.

## Overview

We have migrated from Auth0 (header-based authentication) to BetterAuth (session-cookie-based authentication). This provides a self-hosted, secure authentication layer that eliminates the `x-user-email` header vulnerability.

## Database Architecture

BetterAuth introduces several core tables:

- `auth_user`: Core identity table (BetterAuth managed).
- `auth_account`: Stores credentials (passwords) and OAuth links.
- `auth_session`: Tracks active user sessions.
- `auth_verification`: Handles email/OTP verification tokens.

The legacy `users` table is linked to BetterAuth via the `auth_user_id` column.

Since the frontend registration is restricted (invite-only), several CLI scripts are provided in the `scribe-server` repository for administrative tasks.

## Environment Configuration

To deploy or review this system, the following environment variables are required.

### Backend (`scribe-server/.env`)

- `BETTER_AUTH_SECRET`: A secure 32-character random string used to sign sessions and tokens.
  - **Generation**: You can generate one using `openssl rand -hex 32`.
- `BETTER_AUTH_URL`: The full URL of the auth API (e.g., `http://localhost:9999/api/auth`).
- `FRONTEND_URL`: The URL of your web application (used for CORS and trusted redirects).

### Frontend (`scribe-web/.env`)

- `VITE_BETTER_AUTH_URL`: Should match the backend `BETTER_AUTH_URL`.

### 1. Initial Data Migration

If you are setting up a new environment with existing legacy users, run this to sync them into the BetterAuth system:

```bash
npm run db:migrate-users
```

_This is an idempotent script that creates BetterAuth records for any user missing an `auth_user_id`._

### 2. Creating New Users

To create a brand new user from the CLI (useful for QA testing):

```bash
npm run db:create-user <email> <password> <username> [roleId]
```

- `roleId`: 1 for Manager, 2 for Translator (defaults to 2).
- This creates records in `auth_user`, `auth_account`, and the application `users` table.

### 3. Setting/Updating Passwords

To manually set or override a password for an existing user (e.g., legacy users who migrated but never set a password):

```bash
npm run db:set-password <email> <password>
```

_Note: This uses BetterAuth's internal hashing utility. Do not attempt to manually insert raw bcrypt strings into the database._

## Frontend Integration Notes

- **Session Handling**: The frontend uses `authClient.useSession()` to determine identity.
- **Redirects**: Upon successful login, the application performs a hard redirect (`window.location.href`) to ensuring the fresh session cookie is immediately picked up by the router.
- **Custom UI**: The login and password reset pages are custom-built to match the classic Fluent brand style.

## Technical Implementation

### Global Authentication Middleware

The backend uses a global `authenticate` middleware (in `src/middlewares/authenticate.ts`) that runs on every request:

1. Validates the session via `auth.api.getSession()`.
2. Populates `c.get('session')` with the BetterAuth session.
3. Automatically looks up the application user in the `users` table and populates `c.get('user')`.

Downstream routes use `authenticateUser` (in `src/middlewares/role-auth.ts`) to enforce that a valid application user is present before proceeding.

## Important Technical Details

### Password Hashing

BetterAuth uses a specific salted hashing format. Standard `bcrypt` hashes generated by external tools will fail verification. Always use the `db:set-password` or `db:create-user` scripts to ensure passwords are saved correctly.

### Invitations

The system supports magic links for invitations. This can be triggered via the `magicLink` plugin in the backend `src/lib/auth.ts`.
